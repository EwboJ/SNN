# 走廊数据失衡分析与稀疏化处理方案
> 分析日期: 2026-03-09 16:02 | 排除 left4_bag2(36帧异常)

---

## 1. 动作分布统计 (3类映射)

| 动作 | 帧数(估算) | 占比 | 失衡倍率 |
|---|---:|---:|---:|
| **Straight** (Forward+Stop) | ~7,500 | **~58%** | 2.8× |
| **Left** | ~2,800 | **~22%** | 1.0× (基准) |
| **Right** | ~2,500 | **~19%** | 0.9× |
| **合计** | ~12,848 | 100% | — |

### 失衡可视化
```
Straight: ████████████████████████████████████████████████████████████ 58%
Left:     ██████████████████████                                     22%
Right:    ███████████████████                                        19%
```

> [!WARNING]
> **Straight 类是 Left/Right 的约 2.8 倍**，存在明显的类别失衡。
> Left 和 Right 之间相对均衡(22% vs 19%)，这是好的。

---

## 2. 失衡的根本原因

每个 run 的时间序列为：`Stop(起步) → Forward(直行) → Left/Right(转弯) → Forward(出弯)`

- 直行段通常占整个 run 的 **55-65%**（机器人从走廊中间走到拐角处需要较长直行距离）
- 转弯段只有 **35-45%**（转弯动作本身持续时间短）
- 这是**结构性失衡**：采集流程决定了直行帧天然偏多

---

## 3. 是否需要稀疏化处理？

### 3.1 什么是稀疏化（Temporal Downsampling）

稀疏化 = 对连续帧进行**隔帧采样**，降低连续同类帧的冗余度。

### 3.2 分析：是否需要

| 因素 | 分析 | 结论 |
|---|---|---|
| **帧间相似度** | ~30 FPS 下相邻帧几乎一样（尤其直行段） | ⚠️ **存在严重冗余** |
| **直行帧冗余** | 连续 100+ 帧的 Forward 场景几乎相同 | ⚠️ **最严重** |
| **转弯帧冗余** | 转弯中场景变化较快 | ✅ 相对较少 |
| **训练效率** | 冗余帧浪费训练时间但不增信息量 | ⚠️ **需要处理** |
| **SNN 特殊性** | SNN 可利用时序信息（framediff 模式） | ✅ 保留时序有价值 |

> [!IMPORTANT]
> **结论：需要对 Straight 类进行稀疏化，但不能简单删帧——需要分情况处理。**

### 3.3 推荐方案

#### 方案 A: 在 DataLoader 层面用 `weighted_sampler`（推荐，最简单）

**不删帧，通过采样权重解决失衡。**

- 每个 epoch 中 Left/Right 帧被采样的概率 = Straight 帧的 ~2.8 倍
- 等效于对少数类做过采样、多数类欠采样
- **优点**: 不丢弃数据、代码已实现、可随时调整
- 训练命令: `--class_balance weighted_sampler`

```python
# 等效采样权重
weights = {
    'Straight': 1.0,    # 基准
    'Left':     2.68,   # 7500/2800 ≈ 2.68
    'Right':    3.00,   # 7500/2500 ≈ 3.00
}
```

#### 方案 B: 对 Straight 帧做预处理降采样

在数据预处理阶段对连续 Forward 段做**每 N 帧取 1 帧**：

```python
# 伪代码：对连续 Forward 段做 1/3 降采样
if 连续 Forward 段长度 > 10:
    只保留每第 3 帧
else:
    全部保留  # 转弯边界附近不降采样
```

降采样后的预期分布：
| 动作 | 原始 | 降采后(1/3) | 占比 |
|---|---:|---:|---:|
| Straight | ~7,500 | ~2,500 | 32% |
| Left | ~2,800 | ~2,800 | 36% |
| Right | ~2,500 | ~2,500 | 32% |
| 合计 | ~12,848 | ~7,800 | — |

- **优点**: 真正消除冗余，训练更快
- **缺点**: 需要新写预处理脚本，且可能丢失直行-转弯过渡帧

#### 方案 C: CE Loss 类别权重

在损失函数中对少数类加权：

```python
# CrossEntropyLoss with class weights
class_weights = torch.tensor([
    1.0,   # Left (基准)
    0.37,  # Straight (2800/7500 = 0.37)
    1.12,  # Right (2800/2500 = 1.12)
])
loss = CrossEntropyLoss(weight=class_weights)
```

- 训练命令: `--class_balance class_weight`

## 4. 推荐组合策略

> [!TIP]
> **最佳实践：方案 A + 方案 B 组合使用**

### Step 1: 对 Straight 帧做温和降采样 (删冗余但保留转弯过渡)

对每个 run 中的**连续 Forward 段**做 **每 2 帧取 1** (即 15 FPS)：
- 转弯帧（Left/Right）和 Stop 帧全部保留
- 转弯前后各保留 5 帧不降采样（保留过渡区域）
- 降采后总帧数 ≈ **~9,000**

### Step 2: 训练时使用 `weighted_sampler`

消除降采后残余的失衡：

```bash
python train.py --dataset corridor --corridor_root ./data/corridor \
    --mode discrete --action_set 3 \
    --class_balance weighted_sampler \
    ...
```

### Step 3: 训练两组实验对比

| 实验 | 处理 | 目的 |
|---|---|---|
| Exp-A | 原始数据 + weighted_sampler | 基线 |
| Exp-B | 降采后数据 + weighted_sampler | 去冗余效果 |
